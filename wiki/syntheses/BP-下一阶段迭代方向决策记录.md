# BP 下一阶段迭代方向决策记录

状态日期：2026-07-10。性质：`design_discussion_in_progress_non_runtime`。

本文记录 `grill-with-docs` 讨论过程，用来决定 BP 系统在完成 compile-first、中立事实召回和可审计输出之后，下一阶段优先迭代什么。它是活的维护者 synthesis，不是 runtime 规则；只有讨论中被明确采纳的结论，才会进入对应 skill references 和实现计划。

背景复盘见 [[syntheses/BP-知识压缩与决策质量演进复盘|BP 知识压缩与决策质量演进复盘]]。

## 讨论目标

我们要确定的不是“还能加什么功能”，而是下一阶段唯一的首要里程碑：哪项工作最能提高真实 BP 质量，并且能让我们判断改动是正向还是再次发生偏移。

## 当前已知事实

- 104 个 BP-active 英雄和 Ranked 地图已经有结构化稳定事实。
- strength profile 已作为独立版本输入，不应重写能力和地图事实。
- runtime index 已经可再生，模型可通过 query / hydrate 小窗口消费事实。
- 工具已收窄为中立事实召回，最终判断重新归还给 LLM。
- 对局报告与 decision log 已分层，原则上可以逐回合审计。
- 最近几轮改动多次在“强度优先、地图能力优先、counter 优先”之间摆动，说明缺少稳定的质量判定闭环。
- 生成式 BP 不适合用“某局必须选择唯一英雄”的脆弱红测锁死；但完全不做稳定评估，又无法分辨优化与偶然样本。

## 讨论进展 1：关系覆盖与用户英雄池

用户提出两条相互独立但可以并行设计的方向。

### 质量方向：让决策看到更全面的关系

用户对高质量 BP 的初步理解不是“命中唯一答案”，而是候选与当前局面之间存在更全面的有效关系：

- 先手英雄应与地图有强、稳定的关系，即使对方后续能 counter，也不会轻易失去基础作用。
- 后手候选有时主要放大队友，有时主要惩罚对手；两种价值都可能成立。
- 当对方阵容均衡、短板不明显时，不应强行寻找 punish，而应优先让己方补位、掩护和战线维持更完整。
- 职业赛场常见的保守对线阵容，价值来自队友之间的互补关系，不一定来自对敌方某个英雄的明显 counter。

这修正了此前“质量评估闭环优先”的单一路线。评估闭环仍然必要，但它更适合作为验证底座；真正需要被迭代的质量对象可能是关系模型本身。

### 产品方向：让决策只推荐用户可执行的英雄池

两名潜在用户都自然地把产品理解为决策辅助，并希望接入自己的英雄数据。当前维护者是全英雄满资源玩家，因此此前默认了完整英雄池，没有把不完整账号当作主要场景。

用户提出的最小产品能力是：读取账号英雄信息，在召回侧遮罩用户不可用的候选。这个方向与当前架构兼容，因为 runtime 已经支持 `candidate_pool`、`known_player_constraints` 和中立 `exclude-id`；缺少的是账号输入和约束归一化，而不是另一套 BP 引擎。

## 当前实现与关系模型的差距

| 关系类型 | 当前承载方式 | 当前缺口 |
| --- | --- | --- |
| 英雄—地图 | `map_hooks`、matched capabilities、objective contracts | 已是一等事实，但需要继续清理假阳性和成立条件。 |
| 英雄—对手 | `conditional_matchups` 编译为 `matchup_index` | 已是一等关系，但部分来源只有“有利”标签，缺机制和失效条件。 |
| 英雄—队友 | capability tags、objective contracts、`needs_teammate_support` 由 LLM 临时组合 | 不是一等关系；缺少稳定的 enable、cover、tempo、resource、formation 等关系表达。 |
| 英雄—用户账号 | `candidate_pool` / `known_player_constraints` / `exclude-id` | 召回接口已具备；缺账号读取、拥有/资源/熟练度分层和产品模式。 |

因此，“后手为什么总在局部回答对面，而不放大己方组合”不完全是 prompt 问题：现有 index 对敌方有显式关系边，对队友却主要只有零散能力事实。两边信息密度不对称，LLM 自然更容易看见 counter，而不容易稳定看见补位与掩护。

## 术语澄清

### 关系边数量不是合适的北极星

单纯追求更多边会重演“数据越厚越好”的问题。一条没有机制、成立条件或失效条件的假边，比缺一条边更危险。更精确的候选术语是：

`决策相关关系覆盖`：在当前地图、slot、己方阵容和敌方阵容下，系统能召回多少条已被局面激活、机制完整、方向互补的关系。

一个候选的 `activated relation portfolio` 可以包含：

- `map_anchor`：候选在地图和模式中的基础职责。
- `ally_enable`：候选让队友更容易完成其职责。
- `ally_cover`：候选修补队友的暴露和失败条件。
- `enemy_exploit`：候选惩罚敌方已公开的路线或结构。
- `enemy_exposure`：敌方已公开英雄如何反过来限制候选。
- `response_robustness`：对方还有多少低成本方式让候选失去主要价值。

先手安全性不能只由一条强 `map_anchor` 推出。更完整的判断应是：地图基础职责稳定，同时对常见后续回应具有足够 `response_robustness`，且不会迫使后续两手只能围绕它补洞。

### 用户约束不是版本理解

用户输入至少要拆成三层：

| 用户数据 | 含义 | 推荐处理 |
| --- | --- | --- |
| ownership / availability | 是否拥有、是否达到当前玩法的可用门槛 | live BP 中作为硬遮罩。 |
| resource readiness | 等级、Hypercharge、星辉、小工具、装备是否齐全 | 根据英雄构筑依赖决定硬约束或风险提示。 |
| proficiency / comfort | 用户是否擅长或愿意使用 | 作为偏好或证明门槛，不应伪装成版本强度。 |

`strength_profile` 仍表示当前版本环境中的英雄强度。账号英雄池属于 `player_profile` 或 `account_constraints`，两者必须分开，否则会把“用户没有”误写成“英雄不强”。

## 关系边的存储选择

关系模型存在两个极端：

1. 为所有英雄对显式维护 pairwise 边。优点是直接；缺点是组合量迅速膨胀、维护漂移严重，而且容易生成大量空泛边。
2. 完全由 LLM 从能力标签临场推导。优点是轻；缺点是当前已经证明队友协同容易被漏掉，推导也难以审计。

当前建议采用混合模型：

- 英雄特有、机制明确的关系显式维护，例如特定附身、传送、增益、召唤物互动或独特资源循环。
- 通用协同通过有类型的能力关系派生，例如治疗支撑前排、开墙释放长线、加速帮助短手接触、控制保护持续输出。
- 派生关系仍必须输出 mechanism、active_when 和 fails_when，不能只生成 `good_with`。
- compile 可以生成关系视图，但 query 工具仍只返回中立关系事实，不替 LLM 判断哪条关系当前最重要。

## 讨论纠偏 2：要量化的是决策，而不是数据建设

用户明确纠正了讨论对象：Fandom 与 PLP 已作为当前社区最权威的来源完成 raw capture 和 ingest，稳定事实层视为现阶段 ground truth。当前要持续推高和评价的是 ban / pick 决策质量，不是数据生产数量或 source 健康度。

因此，“关系边数量”应理解为一次具体决策中候选与当前局面的关系数量，而不是数据库总边数。

### 决策状态中的关系组合

对于当前状态 `S` 和候选 `C`，可以把候选的关系组合拆成一个向量，而不是混成总分：

```text
R(C, S) = {
  map_anchor,
  objective_contribution,
  ally_enable,
  ally_cover,
  enemy_exploit,
  enemy_exposure,
  activated_failure
}
```

- `map_anchor`：候选与当前地图结构的正关系。
- `objective_contribution`：候选对当前模式胜利条件的直接贡献。
- `ally_enable`：候选放大已选队友的能力或资源。
- `ally_cover`：候选修补已选队友的失败条件。
- `enemy_exploit`：候选惩罚敌方已公开英雄或阵容结构。
- `enemy_exposure`：敌方已公开选项对候选形成的反向关系。
- `activated_failure`：当前地图或阵容已经激活的候选失败条件。

所有计数只接受 runtime index 中有稳定事实引用的唯一 relation id，并且必须与当前 map、mode、己方已选或敌方已选节点相连。报告里多写几句同义理由不能增加计数。

### 可以硬性观察的决策指标

| 指标 | 定义 | 用途 |
| --- | --- | --- |
| `decision_relation_depth` | 最终候选在当前状态中实际成立的唯一关系边数量，正向与负向分开统计 | 观察决策是否建立在足够丰富的局面联系上 |
| `decision_relation_breadth` | map / objective / ally / enemy / exposure / failure 中实际覆盖的关系家族数 | 发现只看强度、只看地图或只看 counter 的单眼决策 |
| `considered_relation_coverage` | decision trace 实际比较的关系边 / 当前 serious candidates 可见的状态相关关系边 | 观察模型是否漏看已经召回的关键关系 |
| `non_dominated_selection_rate` | 最终选择是否处于 serious candidates 的关系组合 Pareto 前沿 | 识别被其他候选全面压过的明显劣选 |
| `hard_failure_rate` | 选中项触发地图/目标 hard gate、无依据关系或明显职责缺失的比例 | 量化不可接受的决策下限 |

其中最关键的是 `non_dominated_selection_rate`。它不要求人为发明权重：

- 如果候选 A 在 map、objective、ally、enemy 正关系上都不少于 B，并且 exposure / failure 不多于 B，至少一项严格更好，则 A 支配 B。
- 如果最终选了被支配的 B，这手可以被硬性判为关系层面的低质量决策。
- 如果 A 更能配合队友、C 更能惩罚对手，两者互不支配，则数字无法替代策略取舍；它们都留在 Pareto 前沿，由 LLM 根据局面和风格选择。

示意：

| 候选 | map/objective | ally | enemy | exposure/failure | 判断 |
| --- | ---: | ---: | ---: | ---: | --- |
| A | 3 | 2 | 1 | 1 | 支配 B |
| B | 1 | 0 | 1 | 3 | 被支配，选出需要额外解释 |
| C | 2 | 3 | 0 | 0 | 与 A 互不支配，属于真实策略选择 |

这给出的是**可量化的偏序**，不是把完整 BP 压成一个标量：

- 代码可以稳定识别关系贫乏、观察不全和被全面支配的选择。
- LLM 只处理 Pareto 前沿内部那些价值方向不同、无法客观全序的候选。
- 决策质量可以因此减少对 LLM 裁判的依赖，但不能被边数完全替代。

### 先手和后手的可观测关系不同

- 先手尚无队友和敌方已选，主要观察 map / objective 关系、可扩展性和对可信回应池的 `response_surface`。地图关系很多但存在大量低成本广谱回答，也不能直接视为安全首手。
- 后手已有公开节点，应观察 ally enable / cover、enemy exploit、enemy exposure 与当前阵容职责。对方没有明显短板时，ally 关系更多的候选可以自然进入 Pareto 前沿，而不是被迫寻找 counter。
- 双选还应观察两个新英雄之间的内部关系，以及它们是否重复解决同一问题却留下共同缺口。

## 先精确定义“BP 质量”

`BP 质量` 不是一个总分，至少包含以下不同判断：

| 维度 | 要回答的问题 | 典型失败 |
| --- | --- | --- |
| 事实可靠性 | 论据是否来自当前地图、英雄和条件化关系事实？ | 捏造 Backyard Bowl 有水，或把来源不足的关系说成确定 counter。 |
| 目标有效性 | 选项能否完成模式胜利条件和该地图的关键职责？ | Heist 选出缺少打库、守库或 race 价值的阵容。 |
| 阵容整体性 | 三个英雄是否形成完整胜利路线，而不是三个局部好选项？ | 每手只回答对面一个英雄，最后己方没有可执行组合。 |
| 对位利用 | 是否识别当前阵容下真正成立的多目标回答或结构惩罚？ | 后手完全忽略公开阵容，或把单点 counter 当硬目标。 |
| 计划连续性 | ban、首手、双选和最后手是否围绕同一可修正计划推进？ | ban 保护了一个计划，首手却改拿无关强度英雄。 |
| 版本强度利用 | 在结构成立的候选中，是否合理利用当前版本强弱？ | 强度创造地图适配，或完全忽略明显超模例外。 |
| 可审计性 | 人能否看清关键论据、替代方案和拒绝原因？ | 输出大量字段或总分，却无法定位哪条证据推动了选择。 |

这些维度不能再合成一个公式总分。下一阶段的评估应保留它们的差异。

## 候选迭代方向

### 方向 A：先建立质量评估闭环

建立一组固定 draft 状态的 `BP audit casebook`，覆盖不同模式、slot 权力和已知失败类型。每个 case 固定公开事实和 runtime index 版本，但不规定唯一正确英雄。

建议评估方式：

- 同一 case 用当前版和候选版分别决策，隐藏版本信息后进行 A/B 盲审。
- 人只选“更可信的一份”或“无法区分”，不计算综合总分。
- 另外记录硬失败标签，例如 `unsupported_map_fact`、`objective_duty_missing`、`local_counter_overfit`、`team_plan_break`、`ban_without_plan`、`relation_without_mechanism`。
- 保存每次比较使用的 index hash、skill 版本、模型、公开 draft 状态、工具召回和 decision log。
- case 可以稳定复跑，但不把某个英雄写成唯一标准答案。

收益：后续任何数据、skill 或 prompt 修改都有比较基线，可以判断到底改善了哪类问题。

风险：需要维护者持续做少量人工判断；它不会立刻让某一局变强，而是先让迭代不再盲目。

### 方向 B：扩展决策关系模型

基于已 ingest 的 Fandom / PLP ground truth，把当前隐含在能力事实中的队友 enable / cover、阵容资源和反向 exposure 编译成可召回关系。

收益：直接补齐当前索引偏向敌方 counter、弱化己方协同的结构缺口。

风险：如果没有 decision relation 指标，新增关系仍可能只让产物变厚，无法证明 BP 选择更全面。

### 方向 C：继续优化选手推理流程

加强 match-scoped 计划记忆、ban 计划、双 pick 组合比较、最后手多目标分析和 final draft review。

收益：最接近用户体感中的“会不会 BP”。

风险：没有基线时容易再次把某一种正确倾向硬编码成目标，重演强度优先或 counter 优先的摆动。

### 方向 D：性能和批量运行

实现多地图外层并行、批量实验配置和统一结果汇总。

收益：降低十局、二十局批量观察的耗时。

风险：只会更快地产生样本，不会自动提高样本质量；如果评估方式仍是随手看几局，吞吐增加可能放大阅读负担。

## 初始主张

建议下一阶段先选方向 A，把“可观察”升级为“可比较”。理由不是评估比决策更重要，而是当前最主要的工程风险已经从“信息取不到”变成“每次修改后无法可靠判断方向是否更好”。

方向 A 不应做成固定答案红测，也不应发明新的质量总分。它应是：固定场景、保留开放选项、A/B 盲审、硬失败标签、完整审计证据。

在这个闭环建立后：

- case 暴露决策关系缺口，就进入方向 B 扩展关系模型。
- case 暴露事实已召回但推理未使用，就进入方向 C 修 skill。
- case 运行成本成为主要瓶颈，再进入方向 D 做并行。

经过本轮讨论，这项主张调整为：评估闭环不必作为独占的产品里程碑，但应伴随关系模型迭代同步建立，用来验证新增关系是否真正改善选择，而不是只增加边数量。

## 待决问题 1：关系覆盖的规范形式

> [!question]
> 是否确认把质量方向的北极星从“关系边更多”收窄为“决策相关关系覆盖”，并采用混合关系模型：英雄特有关系显式维护，通用队友协同从能力关系派生，最终以当前局面实际激活的关系组合进行审计？

## 待决问题 2：是否接受“偏序而非总分”

> [!question]
> 是否接受用关系深度、关系广度、关系观察覆盖率、非支配选择率和 hard failure rate 量化 ban / pick 决策，并以 Pareto 前沿而不是混合总分表达候选高低；只有前沿内部互不支配的取舍才交给 LLM？

### 该问题确定后，下一轮需要继续 grill 的问题

1. 哪些队友关系必须显式维护，哪些可以安全派生？
2. `response_robustness` 是关系事实，还是只能由 LLM 在当前 draft 中判断？
3. 账号遮罩针对“未拥有”还是“未达到资源门槛”；不可用理论最优解是否仍应在复盘模式展示？
4. 产品首先服务 live BP、赛前英雄池规划，还是两者分模式支持？
5. 如何用 A/B case 验证新增关系，而不把关系数量变成新的总分？

## 已决定事项

- `BP 质量` 不压成单一混合分。
- 不完整英雄池是有效用户场景，当前 runtime 的召回遮罩机制可以承接该需求。
- `strength_profile` 与用户账号约束是两个不同领域对象，不应合并。
- “关系边更多”仍需收窄定义；是否正式采用“决策相关关系覆盖 + 混合关系模型”尚待确认。
- Fandom / PLP ingest 后的稳定事实层是现阶段 ground truth；当前评价对象是 ban / pick 决策，而不是数据建设质量。
- 决策质量可以先建立可量化偏序，识别关系观察不足、hard failure 和被其他候选全面支配的选择；是否采纳该模型尚待确认。
