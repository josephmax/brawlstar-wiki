# 条件化对位模型

这页定义本地知识库用于 BP 推演的中间层。它的目标不是立刻给出定量分数，而是让 LLM 在每一轮 Ban Pick 中带着地图、模式、阵容、对位和执行条件进行稳定的定性推理。

文件性质：`runtime_schema`。本页长期存在，用来定义 BP 推理会消费的对象和字段；它不承载版本差分、补丁翻译、临时观察名单或批量 ingest 过程记录。

## 目标

BP 输出不应只回答“选谁”，而应回答：

- 当前地图和模式激活了哪些条件？
- 敌方已经暴露了什么胜利路线？
- 我方阵容缺什么职责？
- 哪些 counter 关系在这张图上成立，哪些不成立？
- 当前最优 pick / ban 是建立计划、补缺口、保护队友，还是破坏对手计划？
- 这一手之后，对方最自然的反制是什么？

这层模型来自 [[sources/User-Note-BP-Reasoning-Intermediate-Layer|用户经验来源摘要: BP 需要条件化推理中间层]]，并接入 [[syntheses/Ban-Pick-问题拆分|Ban Pick 问题拆分]]。如果需要让另一个 LLM 直接执行 BP 推理，应优先读取 [[syntheses/BP-推理DSL规范|BP 推理 DSL 规范]]，再回到本页查对象定义。

## 核心原则

`counter` 不是 `A 克 B`。

更准确的表达是：

> A 在某类地图、模式、阵容和 build 条件下，能用某种机制限制 B 的核心计划。

因此，外部来源里的 `countersThese` / `counteredBy` 只能作为候选边。进入本地 BP 模型前，必须被转译为条件化对位。

## 对象一：map_profile

地图不是名字，也不只是 `open`、`wall_density`、`water_value` 这类粗分档。粗标签不应进入 BP 输入；真正用于 BP 推理的是具体地形特征如何打开路线、关闭路线、制造目标角度、奖励某些能力或产生假阳性适配。

```yaml
map_profile:
  name:
  mode:
  topology:
    zones:
    lanes:
    barriers:
    routes:
  objective_access:
    objective_type:
    direct_damage_lanes:
    base_entry_routes:
  tactical_features:
    - id:
      type:
      location:
      condition:
      rewards_capabilities:
      punishes_capabilities:
      false_positive_capabilities:
      draft_implication:
  lane_dynamics:
    congestion_triggers:
    split_pressure_value:
    grouping_punishers:
  map_rules:
```

这些字段负责激活或关闭对位关系。例如长线开阔图会增强狙击、拆墙和远程消耗；墙多侧路强的图会增强刺客、投掷和近身进场。但更高质量的表达应继续写清楚：是哪条路线被打开、哪个目标角度成立、哪个墙角能让英雄站住、哪个拥挤条件会让连锁或穿透收益上升。

完整地图 schema 见 [[syntheses/BP-地图建模与决策规范|BP 地图建模与决策规范]]。例如 `Safe Zone` 不能只写成 `water_presence: high`，而应区分：

- 河道让 `Angelo`、`Eve` 这类可过水英雄不容易被堵在家里。
- 远程或特殊弹道英雄可能不需要过河就能对金库施压，例如 `Grom`。
- `water_crossing_short_range_without_base_anchor` 是假阳性：能过河但手短、过河后缺少生存锚点或目标压力，仍可能坐牢。
- 敌方不分路时中路拥挤，会抬高连锁、穿透、召唤物、直线控制等能力的价值。
- 金库旁墙角可能让部分英雄过河后在敌方基地附近连续生存。

26 张 Ranked Season 46 地图完成 `bp_map_profile_v2` 后，地图还需要继续派生为 `map_bp_factors`。这层不再问“地图有什么”，而问“这些地图条件在当前 BP 中制造了什么职责、硬门槛、地形状态计划和 slot 风险”。完整表达见 [[syntheses/BP-地图建模与决策规范|BP 地图建模与决策规范]]。

```yaml
map_bp_factors:
  active_objective_contract:
  active_map_bp_factors:
  duty_coverage:
  terrain_state_plan:
  false_positive_alerts:
```

如果没有这层，LLM 容易把地图页读成强势英雄列表；有了这层，地图才能进入 `must_answer`、`must_avoid`、`required_capabilities` 和 `candidate_eval.map_factor_fit`。

## 对象二：mode_objective_profile

模式决定“赢法”。

```yaml
mode_objective_profile:
  mode:
  primary_goal:
  required_roles:
  tempo_pressure:
  comeback_paths:
  objective_damage_importance:
  survival_importance:
  mobility_importance:
  control_importance:
```

例如 `Brawl Ball` 不能只看击杀对位，还要看推进、破门、抢球、得分窗口和加时清墙。`Heist` 则必须强制考虑 safe damage、开图和持续输出。

## 对象三：brawler_profile

英雄不是一个定位标签，而是一组战术能力。

```yaml
brawler_profile:
  name:
  effective_range:
  projectile_reliability:
  burst:
  sustained_dps:
  objective_damage:
  mobility:
  survivability:
  engage:
  disengage:
  anti_aggro:
  anti_tank:
  wall_break:
  throw_or_wall_bypass:
  area_control:
  scouting_or_vision:
  team_support:
  role_tags:
  failure_modes:
```

`failure_modes` 很重要。它记录这个英雄在什么条件下失效，例如慢弹道怕高机动、短手怕长线、投掷怕开墙、坦克怕反坦和持续减速。

## 对象四：build_profile

Gadget、Star Power 和 Gears 改变英雄能力，因此 build 不是装饰。

```yaml
build_profile:
  brawler:
  gadget:
  star_power:
  gears:
  enables:
  mitigates:
  weakens:
  best_when:
  poor_when:
```

例如 `Brock` 的位移类 Gadget 可以缓解 aggro 压力；`Vision` gear 会提高草丛图的信息价值；`Speed` gear 会让草丛地图中的近身英雄更容易激活进场路线。

## 维护规则：版本资料接入门槛

版本补丁、临时 meta、重做、Buffies、Hypercharge 改动不直接进入 BP 运行时。它们先作为来源或审计材料保存；只有当资料改变英雄的能力类型、职责归类、硬门槛、对位成立条件、地图 hook 或 slot 策略时，才直接更新对应英雄页或地图页的稳定 BP 字段；运行时索引由 compile 阶段重新生成。

```yaml
version_audit_item:
  subject:
  source:
  observed_change:
  qualitative_bp_effect:
    affects_hard_gate:
    affects_required_capability:
    affects_matchup_condition:
    affects_map_hook:
    affects_slot_strategy:
  runtime_decision: merge_to_stable_profile | source_only | watchlist
  reason:
  evidence_needed:
```

接入判断的核心不是“加强/削弱”，而是回答：这条资料是否改变了 BP 中可消费的条件。

- 能改变 `must_pick`、`must_ban`、`must_answer` 或 `must_avoid` 的，才可能进入硬门槛。
- 能改变一个英雄在某模式/地图中的职责归类，才进入 `required_capabilities` 或 `map_bp_factors`。
- 能改变对位边的成立条件或失效条件，才进入 `conditional_matchup`。
- 只能说明数值边际、容错变化的资料，可以进入当前 `combat_breakpoint_profile`、patch source manifest 和 `outputs/balance-breakpoints/` 的生成式断点审计，但不因此更新运行时页面。只有断点确实跨档，且补足命中形态、构筑、地图路线、成立/失效条件和 `bp_use` 后，才提升到现有稳定 BP 字段。

## 对象五：conditional_matchup

这是静态 counter 表进入本地知识库后的主要形态。

```yaml
conditional_matchup:
  subject:
  target:
  direction: subject_favored | target_favored | volatile | even
  confidence: structural | map_dependent | comp_dependent | weak_signal
  mechanism:
  active_when:
    map_conditions:
    mode_conditions:
    comp_conditions:
    build_conditions:
  fails_when:
    map_conditions:
    mode_conditions:
    comp_conditions:
    build_conditions:
  bp_use:
    as_first_pick:
    as_response_pick:
    as_last_pick:
    as_ban_reason:
```

这里的 `mechanism` 必须写清楚。不能只说 `Mortis counters Brock`，而要写：

> Mortis 在墙体、草丛、侧路或队友掩护允许接近时，可以绕过 Brock 的射程优势并贴脸击杀；但在长线开阔、过道明确、接近路径单一的地图中，这条对位边会明显减弱。

## 对象六：draft_state

每轮 BP 都要先更新当前局面。

```yaml
draft_state:
  map_profile:
  mode_objective_profile:
  bans:
    our_bans:
    enemy_bans:
  picks:
    our_team:
    enemy_team:
  pick_order:
  remaining_slots:
  revealed_plans:
  role_gaps:
  active_threats:
  protected_picks:
  exposed_picks:
```

`revealed_plans` 用来记录对方已经暴露的阵容方向，例如 safe DPS、双 sniper、坦克推进、投掷控区、刺客收割、治疗前排等。

## 对象七：pick_slot_state

理解 `1-6` 号位时，必须区分全局顺位和队伍视角。常见六手 draft 顺序是：

```text
1
2-3
4-5
6
```

其中 `1`、`4`、`5` 属于同一队，`2`、`3`、`6` 属于另一队。因此 `4-5` 位不是“已知敌方 1-3 位”，而是“已知己方 1 位和敌方 2-3 位”。这个修正来自 [[sources/User-Note-BP-Slot-Perspective|用户经验来源摘要: BP 位置视角修正]]。

```yaml
pick_slot_state:
  global_slot:
  team_side: first_pick_team | second_pick_team
  known_own_picks:
  known_enemy_picks:
  remaining_own_slots:
  remaining_enemy_slots:
  information_role:
  main_job:
  main_risk:
```

按全局顺位可以理解为：

```yaml
slot_1:
  team_side: first_pick_team
  known_own_picks: []
  known_enemy_picks: []
  remaining_enemy_slots: [2, 3, 6]
  main_job: 稳定先手，建立方向，避免被 2-3 位轻易惩罚

slot_2:
  team_side: second_pick_team
  known_own_picks: []
  known_enemy_picks: [1]
  remaining_enemy_slots: [4, 5]
  main_job: 回答 1 位，同时打开己方计划

slot_3:
  team_side: second_pick_team
  known_own_picks: [2]
  known_enemy_picks: [1]
  remaining_enemy_slots: [4, 5]
  main_job: 与 2 位成组，形成第一层结构

slot_4:
  team_side: first_pick_team
  known_own_picks: [1]
  known_enemy_picks: [2, 3]
  remaining_enemy_slots: [6]
  main_job: 回应敌方 2-3 位，并修正己方 1 位路线

slot_5:
  team_side: first_pick_team
  known_own_picks: [1, 4]
  known_enemy_picks: [2, 3]
  remaining_enemy_slots: [6]
  main_job: 补完己方结构，同时避免暴露给 6 位的一手破局

slot_6:
  team_side: second_pick_team
  known_own_picks: [2, 3]
  known_enemy_picks: [1, 4, 5]
  remaining_enemy_slots: []
  main_job: 最终反制，惩罚敌方已经无法修复的结构问题
```

因此，`4-5` 位的核心不是“针对敌方 1-3 位”，而是：在已知敌方 `2-3` 成组回答后，补完己方 `1` 位开出的计划，并尽量不给敌方 `6` 位留下结构性破口。

## 对象八：draft_eval

每一步推演都应输出定性评估，而不是只给单点结论。

```yaml
draft_eval:
  our_win_condition:
  enemy_win_condition:
  activated_matchups:
  disabled_matchups:
  role_coverage:
  exposed_weaknesses:
  execution_difficulty:
  enemy_response_pressure:
  ban_priority:
  top_decisions:
    - action:
      candidate:
      purpose:
      reasons:
      risks:
      likely_enemy_response:
  qualitative_result: favored | even | risky | losing
```

`top_decisions` 是面向 LLM 输出的核心。每一轮 BP 至少应给出几个当前最优决策，并明确每个决策的用途。

## BP 推演流程

完整执行合约见 [[syntheses/BP-推理DSL规范|BP 推理 DSL 规范]]。本页只定义中间层对象和基础流程。

每一步 BP 可以按这个顺序运行：

1. 读取 `map_profile` 和 `mode_objective_profile`。
2. 更新 `draft_state`。
3. 更新 `pick_slot_state`，明确当前手的已知己方、已知敌方和剩余反制位。
4. 判断双方已经暴露的胜利路线。
5. 用地图、模式、阵容和 build 过滤候选 `conditional_matchup`。
6. 找出我方职责缺口和敌方暴露弱点。
7. 生成 pick / ban 候选。
8. 给每个候选标注用途：建立计划、补职责、保护队友、破坏敌方计划、增加对方执行难度。
9. 输出 `draft_eval`，给出 2 到 4 个优先决策和理由。

## Brock 样例

静态说法：

```text
Mortis counters Brock
```

条件化说法：

```yaml
conditional_matchup:
  subject: Mortis
  target: Brock
  direction: subject_favored
  confidence: map_dependent
  mechanism: Mortis 通过突进绕过 Brock 的射程优势，并在贴脸后利用 Brock 慢弹道和低自保完成击杀。
  active_when:
    map_conditions:
      - 墙体密度高
      - 侧路或草丛能缩短接近距离
      - Brock 的射线被遮挡
    comp_conditions:
      - Brock 缺少队友保护
      - Mortis 队友能压低 Brock 或逼出保命资源
  fails_when:
    map_conditions:
      - 长线开阔
      - 过道明确
      - 接近路径单一
      - Brock 能提前消耗和开图
    comp_conditions:
      - Brock 队友有稳定反刺客能力
  bp_use:
    as_response_pick: 只在地图允许接近且敌方缺保护时成立
    as_ban_reason: 如果我方计划依赖 Brock 且地图允许刺客接近，可考虑 ban 或提前补反刺客
```

这个表达能避免把地图依赖关系误写成无条件克制。

## 来源接入规则

外部资料进入本地 BP 模型时按下列方式转译：

- `best modes` 进入 `mode_objective_profile` 和 `brawler_profile` 的适配备注。
- `recommended build` 进入 `build_profile`。
- `countersThese` 和 `counteredBy` 先进入候选 `conditional_matchup`，再补机制、成立条件和失效条件。
- 版本补丁、Buffies、Hypercharge 和临时 meta 先进入来源页或审计页；只有产生定性 BP 影响时，才以当前状态规则直接更新对应英雄页或地图页，并影响 `conditional_matchup.active_when`、`conditional_matchup.fails_when`、`candidate_eval.knowledge_fit` 或 `hard_gate_result`；运行时对位和地图 hook 视图由 compile 重新生成。
- 赛事复盘进入 `draft_eval` 样例，用来校准胜利路线和 last pick 压力。
- 用户经验用于修正模型边界，尤其是模式评价框架和静态 counter 的误用。
- pick 顺位相关经验进入 `pick_slot_state`，用于校准信息状态、反制权和 last pick 压力。
