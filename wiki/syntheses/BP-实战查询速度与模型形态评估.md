# BP 实战查询速度与模型形态评估

状态日期：2026-07-01。性质：`runtime_architecture_note`。

本页回答一个运维问题：当前 Wiki 数据量是否足以支撑实战 Ban Pick 的思考速度，以及运行时更适合通过 skill / 检索查询，还是把知识直接微调进模型。

## 当前数据体量

- `wiki/` 当前约 3.5 MB，444 个 Markdown 页面；`raw/` 约 4.6 MB。
- BP-active 英雄：104 个，全部为 `profile_status: bp_ready`。见 [[syntheses/英雄BP建模执行状态|英雄 BP 建模执行状态]]。
- Ranked Season 46 地图实体：26 张，赛季入口见 [[syntheses/Ranked-Season-46-地图Map-Profile总览|Ranked Season 46 地图 Map Profile 总览]]。
- 运行时索引快照：reviewed 条件化对位边组 396；英雄能力 hook 319；reviewed Ranked 地图 hook 384。
- 现有 skill 检索脚本 `skills/brawl-stars-bp-slot-decision/scripts/bp_index.py` 在样例 `Safe Zone / Brock / Mortis` 与 `Double Swoosh / Gene / Mortis / Max` 上可在约 0.16 秒返回目标地图、英雄页、对位索引命中和地图 hook 命中。

## 判断

当前数据量已经足够支撑实战 BP 的“有依据思考”，但前提是运行时不能全库读取，也不能每手都重新读完所有 runtime 页面。

推荐形态是：`skill + 本地检索 + 预热上下文 + 小范围候选评估`。

不推荐把 Wiki 知识直接当作微调主载体。微调可能让模型输出风格更稳定、步骤更熟，但不适合承载会变化的英雄、地图池、版本资料、对位边和 build 条件。BP 知识的关键价值在于可更新、可引用、可审计，而这些正是检索层比微调更强的地方。

## 为什么实时 skill / 检索更合适

实战 BP 每一手真正需要的材料很少：

- 当前地图实体页。
- 当前模式目标和地图职责。
- 已知我方 / 敌方英雄页。
- 当前候选的英雄页。
- 条件化对位边索引中的命中行。
- 地图 hook 索引中的命中行。

这和 [[syntheses/BP-推理DSL规范|BP 推理 DSL 规范]]、[[syntheses/条件化对位模型|条件化对位模型]]、[[syntheses/Ban-Pick-问题拆分|Ban Pick 问题拆分]] 的设计一致：先把局面转成 slot 任务、hard gate、required capabilities，再召回 2-4 个可解释候选。

微调的主要问题：

- 更新慢：版本、地图池、英雄改动需要重新训练或额外覆盖。
- 可追溯弱：很难说明某个判断来自哪条英雄页、地图页或对位边。
- 容易混入旧 meta：模型会把旧版本强度当成记忆，而不是受控运行时事实。
- 不适合条件化对位：`A counters B` 必须看地图、模式、slot、build 和失效条件；这类判断更适合结构化检索后现场组合。

## 速度瓶颈

目前的瓶颈不在磁盘读取，也不在数据量本身，而在两件事：

1. 是否把检索范围限制到当前局面相关页面。
2. LLM 是否每手都重新解释整套 DSL。

如果每次完整读取两个大索引和所有必读 runtime 页面，实战会偏慢。如果做上下文预热，首轮加载 DSL、地图池和当前地图；后续每手只增量读取新英雄页和索引命中，速度可以满足实战讨论。

## 推荐运行策略

- 开局预热：读取 BP DSL、条件化对位模型、地图因素规范、当前地图实体页和模式职责。
- 每手增量：只读取新出现英雄页、候选英雄页、相关对位命中和地图 hook 命中。
- 候选先行：先由索引召回 6-10 个可能候选，再由 LLM 按 hard gate / slot exposure 压到 2-4 个输出。
- 保留引用：最终推荐必须能指回地图页、英雄页、对位边或地图 hook。
- 微调只做流程：若将来有大量高质量 BP 复盘样本，可微调“如何按 DSL 输出”，但不要把事实库本体微调进去。

## 结论

对当前规模，实时 skill / 检索比直接微调更适合，也更可维护。

如果目标是 15-30 秒内完成一手 BP 讨论，当前数据结构已经够用；重点是优化检索路径和上下文缓存。如果目标是 2-5 秒内给出近似自动化推荐，则需要在现有 Markdown 之外增加预编译 JSON / SQLite / 向量或倒排索引，并让脚本承担更多候选过滤；微调不是首选解法。
